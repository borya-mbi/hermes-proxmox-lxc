# Робочий процес (Workflow) для спільної розробки в середовищі DEV

Цей документ описує ідеальний процес розробки (DevOps workflow), який ефективно поєднує **оперативну швидкість** доступу через `su - dev` та **гарантовану надійність** через Git-гібридний підхід.

Цей процес забезпечує, що ви можете швидко тестувати ідеї в єдиному середовищі розробки (`dev`), але при цьому нічого не зламаєте в основному продукті, а нові фічі потраплять до робочих користувачів (наприклад, `user1`, `user2`) тільки після перевірки.

---

## Загальна концепція робочого процесу

Вся робота ділиться на 4 ключові етапи:
1. **Швидкі експерименти (Hot-Dev):** Інтерактивна робота від імені `dev` через `sudo`.
2. **Фіксація (Commit):** Збереження працюючого коду в окрему Git-гілку репозиторія.
3. **Стабілізація та Затвердження (Pull Request):** Перевірка та перенесення коду з гілки `dev` в основну гілку (`main`).
4. **Реліз (Upgrade):** Оновлення робочих середовищ кінцевих користувачів.

---

## Етап 1: Оперативна розробка та тестування

Спочатку ми даємо свободу розробникам (чи ботам-асистентам) швидко змінювати конфіги у спільному середовищі `dev`.

1. **Доступ:** Користувач заходить на сервер через SSH і перемикається на користувача `dev` за допомогою команди:
   ```bash
   sudo su - dev
   ```
2. **Розробка:** Ви знаходитесь у директорії `/home/dev/` (або в місці базування агента). Ця директорія є клоном Git-репозиторія, але переключеним на гілку розробки (наприклад, `dev` або `feature-branch`). Ви вільно редагуєте конфігураційні файли, додаєте нові скрипти чи змінюєте логіку.
3. **Миттєве тестування:** Ви одразу перезапускаєте службу користувача `dev` для перевірки внесених змін:
   ```bash
   sudo systemctl restart hermes-agent@dev
   journalctl -u hermes-agent@dev -f
   ```
4. **Легке відновлення:** Якщо ви випадково пошкодили конфігурацію і сервіс не стартує, вам не потрібно панікувати чи шукати, який саме рядок був змінений хвилину тому. Ви просто виконуєте:
   ```bash
   git restore . # Скасовує всі незакомічені зміни
   sudo systemctl restart hermes-agent@dev
   ```
   Після цього середовище `dev` знову стабільно працює як раніше.

---

## Етап 2: Фіксація результатів (Committing to Git)

Коли ви добилися того, що новий функціонал працює стабільно і без помилок у тестовому середовищі `dev`, ці результати необхідно зафіксувати.

1. Перебуваючи під користувачем `dev` (`sudo su - dev`), ви додаєте змінені файли до Git:
   ```bash
   git add <змінені_файли>
   git commit -m "feat: додано новий функціонал для тестування"
   git push origin dev
   ```
2. Тепер ці зміни надійно збережені на віддаленому Git-сервері (GitHub/GitLab/Gitea) у гілці `dev`. Навіть якщо локальний LXC-контейнер з якихось причин буде видалено, ваша праця та історія розробки залишаться у безпеці.

---

## Етап 3: Стабілізація (Pull Request)

Зміни успішно працюють у `dev`, але ми не можемо напряму перекинути їх робочим користувачам (`user1`, `user2`), оскільки це потенційно небезпечно для стабільності їхньої роботи.

1. У веб-інтерфейсі Git-платформи створюється **Pull Request** (Запит на злиття) з гілки `dev` (чи конкретної `feature`-гілки) у головну гілку `main`.
2. Відбувається **Code Review** (перевірка коду). Інші розробники або боти можуть переглянути зміни та внести свої корективи.
3. Лише після схвалення 코드 натискається кнопка **Merge**, і код вливається в гілку `main`. Завдяки цьому правилу гілка `main` завжди містить **лише стабільний, протестований і фіналізований код**.

---

## Етап 4: Розгортання (Upgrade) для робочих користувачів

Тепер, коли новий функціонал знаходиться в базі (`main`), робочі користувачі можуть отримати ці оновлення. Операційна діяльність користувачів `user1` та `user2` не переривається під час попередніх трьох етапів.

1. Робочі користувачі не редагують файли локально. Вони (або система автоматизації CI/CD) просто запускають скрипт оновлення (наприклад, `scripts/upgrade_prod.sh`).
2. Цей скрипт автоматизує розгортання:
   - Перемикається на відповідного користувача (наприклад, `su - user1`).
   - Затягує останній стабільний код: `git pull origin main`.
   - Автоматично перезапускає сервіс: `sudo systemctl restart hermes-agent@user1`.

---

## Переваги цього підходу

*   **Єдине джерело правди (Single Source of Truth):** Уся конфігурація та інфраструктура керована через Git.
*   **Відсутність конфліктів прав у Linux:** Завдяки використанню `sudo su - dev` усі тестові файли створюються від імені одного власника (`dev`), уникаючи проблеми «файл належить користувачу 1 і заблокований для користувача 2».
*   **Повна ізоляція експериментів:** Ламати та тестувати можна без страху, бо це ніяк не впливає на стабільну поточну роботу користувачів `user1` чи `user2`.
*   **Надійний відкат (Rollback):** У разі фатальної помилки завжди працюють стандартні команди `git restore` та `git revert`.
*   **Контрольований реліз:** Зміни потрапляють до робочих профілів виключно через процес Pull Request, що є стандартом індустрії.
