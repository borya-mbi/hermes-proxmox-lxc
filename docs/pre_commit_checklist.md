# Чек-лист перед публікацією на GitHub (Pre-commit / Pre-push Checklist)

Цей документ містить ключові правила та знахідки щодо безпечної публікації інфраструктурного коду. Якщо ви готуєтеся зробити код відкритим (Open-Source) або завантажити його до віддаленого репозиторію, перед виконанням команд `git commit` або `git push` пройдіться по наступних пунктах.

## 1. Захист чутливих даних (Секрети та Ключі)
Критично важливо не "засвітити" реальні API-ключі від AI-провайдерів або паролі.
- **Пошук за ключами**: Проведіть швидкий пошук у редакторі або терміналі на наявність рядків, що починаються з:
  - `sk-ant-` (Anthropic / Claude-ключі)
  - `sk-` (OpenAI / OpenRouter-ключі)
  - `AIzaSy` (Google Gemini-ключі)
- **Правило шаблонів (.example)**: У репозиторій повинні потрапляти лише шаблони з суфіксом `.example` (наприклад, `deploy/config.yaml.example`). В середині шаблонів усі значення мають бути визначені як змінні з безпечними іменами (напр., `${GOOGLE_API_KEY_1}`).

## 2. Анонімізація мережевої інфраструктури (Домени та IP)
Реальні домени (наприклад, ваші робочі домени `my-company.com` тощо) та зовнішні статичні IP-адреси можуть стати ціллю для сканування.
- **Стандартизовані домени**: Використовуйте `example.com` або `agent.example.com` як плейсхолдери у всіх `.md`- та `.example`-файлах замість ваших бойових доменів.
- **Тестові IP-адреси**: Для демонстрацій у документах замість бойових публічних IP використовуйте діапазон *TEST-NET-3* за стандартом RFC 5737. Наприклад: `203.0.113.12`. Такі IP-адреси є 100% безпечними для публікації.
- **Приватні мережі**: Локальні підмережі (наприклад, `192.168.10.x` або `10.0.0.x`) є цілком безпечними для відображення в публічному описі структури Proxmox.

## 3. Надійна робота `.gitignore`
Конфігурації, які містять справжні секрети та робочі дані (наприклад, логіни до бази даних або приватні ключі SSH), завжди повинні ігноруватись. Переконайтеся, що ваш `.gitignore` охоплює:
- Всі робочі `.env` файли (напр., `deploy/hermes-user.env`, `deploy/hermes-lxc.env`).
- Локально налаштовані `config.yaml`.
- Усі приватні та публічні `authorized_keys`.
- Робочі таблиці `.tsv`, що акумулюють базу поточних розгортань.

## 4. Очищення тимчасових та "Draft" файлів
Під час розробки ви часто створюєте файли з робочим статусом. Перед комерційним комітом переконайтесь, що ви прибрали:
- Внутрішні плани реалізації (напр., `FLASH_IMPLEMENTATION_PLAN.md`).
- Нотатки `TODO.md` (крім тих, які потрібні для спільного трекінгу задач GitHub).
- Дампи логів `*.log` чи результати локального тестування команди `curl`.

## 5. Перевірка кодування файлів (UX/UI стандарти)
- **Em-Dashes**: Переконайтеся, що ви не використовували довге "—" форматування у `.yaml`- та `.sh`-файлах, яке може "випадково" зламати конфігурацію парсерів.
- **BOM (Byte Order Mark)**: При редагуванні на Windows (зокрема, через PowerShell) файли можуть отримати мітку BOM у кодуванні UTF-8. Відправка bash-скрипта (з `#!/usr/bin/env bash`) з BOM-маркером зламає запуск `bash` у контейнері Linux. Редагуйте код у VS Code (UTF-8, LF) або через вбудовані утиліти заміни (sed / awk), а не PowerShell Set-Content.
