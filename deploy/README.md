# Локальні файли деплою

У каталозі `deploy/` зберігаються локальні робочі файли для деплою Hermes. Саме тут змінюються реальні IP, домени, токени, SSH-ключі та службові нотатки.

## Які файли тут є

- `hermes-lxc.env.example` -> копіюється в `hermes-lxc.env`
- `hermes-user.env.example` -> копіюється в `hermes-user.env` **(v2: спільний bootstrap-шаблон)**
- `config.yaml.example` -> копіюється в `config.yaml` **(v2: провайдери та моделі)**
- `authorized_keys.example` -> копіюється в `authorized_keys`
- `port-forwards.tsv.example` -> копіюється в `port-forwards.tsv`
- `npm-proxy-hosts.tsv.example` -> копіюється в `npm-proxy-hosts.tsv`
- `container-notes.tsv.example` -> копіюється в `container-notes.tsv`

Файли без суфікса `.example` ігноруються в git, тому сюди безпечно підставляти реальні параметри.

### Особливості Onboarding Flow (v2)
Файл `deploy/hermes-user.env` **не є** індивідуальним файлом конфігурації. Це спільний шаблон, який копіюється в контейнері в `~/.hermes/.env` при створенні нового користувача. Під час першого розгортання адміністратор запускає спільний канал. Далі, щоб розвести доступ, необхідно для кожного користувача відредагувати його локальний `~/.hermes/.env` безпосередньо в контейнері і запустити `sync_configs.sh`. 

## Що створюється автоматично

Під час роботи скриптів також з'являються службові файли:

- `container_info.tsv` - зведення по розгорнутих контейнерах;
- `failed_ctids.txt` - список CTID, які завершилися помилкою під час деплою.

Ці два файли не копіюються з `.example`, а створюються під час реального запуску.

## Домени і приклади

У прикладах цієї директорії використовується домен `example.com` і тестовий публічний IP `203.0.113.12`. Це реалістичніші значення для читання статті та перевірки шаблонів, але перед реальним запуском їх треба замінити на власні.

Для покрокового розширення доменних записів краще використовувати:

```bash
bash scripts/generate_default_domains.sh --append <ctid>
```

Це додає записи виду `<user>-agent-<ctid>.example.com` і `<user>-adm-<ctid>.example.com` для кожного користувача, без перезапису вже наявних записів.

## SSH-ключі

`authorized_keys` - локальний файл для керування доступом. 
Важливо: цей файл працює як **authoritative desired state** (джерело істини). Якщо цей файл порожній або ви видалите з нього ключі, при наступному оновленні через `update_hermes.sh` ці ключі також **будуть видалені** з `/root/.ssh/authorized_keys` всередині контейнера. Для входу по SSH контейнер повинен мати встановлений `openssh-server`.
